# Automation Solutionz / ZeuZ QA Kit

Generated by ZKME for `OC`.

## Purpose

This artifact makes Automation Solutionz / ZeuZ a first-party automation target for any AI tool working in this repo.

ZKME should not copy the entire ZeuZ Node runtime into every scanned repository. Instead, it installs the repo-local QA structure, evidence packs, action references, and ZeuZ-compatible export locations. ZeuZ Node remains the canonical runtime; ZeuZ Server remains the optional visual management and reporting destination.

## Local Without ZeuZ Server

Generated tests should be able to run locally without ZeuZ Server when the repo has enough runnable evidence. Use `qa/automation/local-runner/zkme-portable-testcase.schema.json` for portable local test cases.

Run portable tests with `qa/automation/local-runner/run_zkme_tests.py`. Local runs should write ordinary local reports under `qa/automation/reports/local/` plus a ZeuZ-compatible envelope under `qa/automation/reports/zeuz-compatible/` using `zeuz-report.schema.json`. That report can later be mapped into ZeuZ Server.

Run the feature QA gate with `qa/automation/verify_zkme_qa_contract.py`. In CI, pass changed files with `--changed-files-from`; locally, pass repeated `--changed-file` values. The verifier makes missing QA evidence fail-fast instead of remaining an advisory instruction.

## Same Test Case In Playwright

Yes: the same generated testcase can run in Playwright. The recommended source of truth is a portable testcase under `qa/automation/local-runner/test-cases/`. A local runner executes `playwright` steps through a Playwright adapter. When developers want normal Playwright ergonomics, ZKME may also export a Playwright spec from that same portable testcase with `qa/automation/local-runner/export_playwright_spec.py`, but the portable testcase remains canonical.

## ZeuZ Report Bridge

ZeuZ Node uploads execution data through `create_report_log_api/` using the `CommonUtil.all_logs_json` shape. ZeuZ Server then updates `test_case_results` and `test_step_results` from `json_data[].test_cases[].execution_detail` and `json_data[].test_cases[].steps[].execution_detail` through `/api/v1/tasks/update-test-results`. ZKME local reports must preserve that projection as `zeuzExecutionLog`.

## Function Extraction

Use `ZEUZ_FUNCTION_CATALOG.md` and `static/zeuz_function_catalog.v1.json` as the first-party capability map. It is extracted from ZeuZ Node action declarations when available and covers Web, Mobile/Appium, REST/API, Database, Desktop, Performance, Security, XML, utility, and common actions.

The target repo structure should be ZKME-native and Playwright-first. ZeuZ functions are source material for normalized adapters; they should not force the target repo to inherit ZeuZ Node's internal layout.

## ZeuZ Node Runtime Install

ZKME must always know how to obtain ZeuZ Node for extraction or direct runtime fallback. Prefer the local sibling checkout at `A:/git/Zeuz_Python_Node` when available. Otherwise use the official open-source dev zip: https://github.com/AutomationSolutionz/Zeuz_Python_Node/archive/refs/heads/dev.zip. Target repos should use `qa/automation/vendor/zeuz-python-node/fetch_zeuz_node.py` only when they need a full vendored runtime.

## Multi-Language Setup

Python targets can call the bridge most directly. TypeScript/JavaScript should use native Playwright for web and call the Python ZeuZ bridge for DB/mobile/desktop/cross-surface actions until native wrappers exist. Java and C# should use the same portable testcase JSON and report format with language-native test runners.

## Hard AI Behavior

- For every feature change, create or update a manual test case.
- Create or update unit tests for logic, validation, parsing, state, permissions helpers, or utilities.
- Create or update API/integration tests for routes, contracts, auth, integrations, workers, queues, or DB behavior.
- Create or update Playwright E2E tests for critical web workflows.
- Create ZeuZ-compatible candidates for cross-surface Web/API/DB/Mobile/Desktop/Performance/Security flows.
- Make local execution possible without ZeuZ Server when the repo has enough runnable evidence.
- Prefer one portable testcase as the source of truth; run it through a Playwright adapter locally and export Playwright specs only when useful for developer ergonomics.
- Write ZeuZ-compatible report envelopes for local runs when generated tests execute.
- Include a ZeuZ execution-log projection in local reports so the result can later map into ZeuZ Server result tables.
- Run the QA contract verifier for feature changes so missing manual/unit/API/web/DB/mobile/desktop/portable/ZeuZ evidence is caught locally or in CI.
- Mark low-confidence automation as a manual candidate instead of writing brittle tests.

## ZeuZ Compatibility

Use ZeuZ-compatible candidates when workflows combine surfaces such as Web, REST/API, Database, Mobile, Desktop, Performance, or Security. Store drafts under `qa/automation/zeuz/` and keep them reviewable before any future ZeuZ Server sync.

The local ZeuZ Node source is expected at `A:/git/Zeuz_Python_Node` in the owner workspace. Public source: https://github.com/AutomationSolutionz/Zeuz_Python_Node
