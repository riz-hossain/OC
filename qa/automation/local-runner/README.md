# ZKME Local Runner

This folder is for portable test cases that can run on a local machine without ZeuZ Server.

The intent is:

```text
ZKME portable testcase
  -> local adapters execute Playwright/API/DB/mobile/desktop steps
  -> optional Playwright spec export is generated from the same testcase
  -> local report is written
  -> ZeuZ-compatible report envelope with zeuzExecutionLog can be exported later
```

## Why This Exists

ZeuZ should remain the first-party visual automation platform, but developers and AI agents must be able to run generated tests locally without a ZeuZ Server dependency.

## Structure

- `test-cases/`: portable JSON test cases generated by AI agents
- `../testcases/`: canonical testcase packages with portable, Playwright, and ZeuZ action implementations
- `run_zkme_tests.py`: local runner for portable testcases
- `export_playwright_spec.py`: optional exporter from portable testcase JSON to Playwright spec files
- `PORTABLE_TESTCASE_EXAMPLES.md`: authoring examples for Playwright, API, DB, and mobile candidates
- `adapters/`: repo-local helper adapters and language shims
- `adapters/zeuz_bridge/`: thin bridge for ZeuZ-derived action rows, function calls, variables, status mapping, and report projection
- `adapters/python/`, `adapters/typescript/`, `adapters/java/`, `adapters/csharp/`: language-specific runners or wrappers when the repo prefers that stack
- `zkme-portable-testcase.schema.json`: schema for local portable test cases

## Execution Contract

A portable local testcase may include:

- `playwright` steps for web UI
- `api` steps for HTTP/GraphQL contracts
- `db` steps for approved local/containerized DB checks
- `mobile` or `desktop` candidates when the repo supports them
- `zeuz` mapping metadata for future import or report sync

Local execution must write results under `qa/automation/reports/`, and ZeuZ-compatible output under `qa/automation/reports/zeuz-compatible/`.

Run all portable testcases:

```bash
python qa/automation/local-runner/run_zkme_tests.py
```

Run one testcase:

```bash
python qa/automation/local-runner/run_zkme_tests.py qa/automation/local-runner/test-cases/SAMPLE_PORTABLE_TESTCASE.json
```

Outputs:

- local JSON report: `qa/automation/reports/local/`
- ZeuZ-compatible report: `qa/automation/reports/zeuz-compatible/`

Export Playwright specs from portable testcases:

```bash
python qa/automation/local-runner/export_playwright_spec.py
```

Output:

- Playwright specs: `qa/automation/web/playwright/generated/`

For web tests, the canonical source is the portable testcase. The runner may execute it directly through a Playwright adapter, or export a normal Playwright spec for developer ergonomics. Do not let the exported spec drift away from the portable testcase.

For mobile, desktop, DB, REST/API, performance, security, XML, and common utility actions, prefer ZeuZ-derived functions from `.zkme/kme_output/static/zeuz_function_catalog.v1.json` and place thin wrappers under `adapters/`.

Use `qa/test-data-matrix.json`, `qa/traceability-matrix.json`, `qa/test-design-quality.json`, and `qa/manual/test-cases/lead-qa-generated/` to decide whether a local portable testcase has enough API, DB, UI/mobile/desktop, auth, negative, fixture, cleanup, traceability, and report evidence to be considered complete.

## ZeuZ Report Projection

When a local testcase runs, the ZeuZ-compatible report should include `zeuzExecutionLog`, a projection of the ZeuZ Node `CommonUtil.all_logs_json` report shape. ZeuZ Server can later map:

- testcase results from `zeuzExecutionLog[].test_cases[].execution_detail`
- step results from `zeuzExecutionLog[].test_cases[].steps[].execution_detail`
- artifacts from the local report paths or future ZeuZ artifact import

## Rule For AI Agents

When a generated test is meant to run locally without ZeuZ, write or update a portable testcase here and map each step to the closest ZeuZ action concept where possible.
