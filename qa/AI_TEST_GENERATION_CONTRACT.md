# AI Test Generation Contract

Generated by ZKME for `OC`.

This repo is AI-ready for feature work. When an AI agent implements or changes a feature, it must create or update test evidence by default.

## Required For Every Feature Change

1. Read `qa/feature-architecture-map.json`, `qa/workflow-map.json`, `qa/test-oracles.json`, `qa/risk-matrix.json`, `qa/test-data-matrix.json`, `qa/traceability-matrix.json`, and `qa/test-design-quality.json` when present.
2. Add or update a lead-QA manual test case under `qa/manual/test-cases/` or `qa/manual/test-cases/lead-qa-generated/`.
3. Cover UI/mobile/desktop, API, DB/data, auth/role, audit/report/log, async, integration, negative, boundary, cleanup, and artifact validation whenever the model marks them applicable.
4. Add or update unit tests when business logic, parsing, validation, state, or utility behavior changes.
5. Add or update API/integration tests when routes, service contracts, data contracts, auth, integrations, workers, or database behavior changes.
6. Add or update Playwright E2E tests, or a clear E2E candidate, when a critical web workflow changes.
7. Add mobile, desktop, DB, performance, or security test candidates when those surfaces are touched.
8. Add a ZeuZ-compatible cross-surface candidate when the workflow combines Web, API, DB, Mobile, Desktop, Performance, or Security actions.
9. When enough runnable evidence exists, keep local execution possible without ZeuZ Server.
10. When a generated local test runs, write or preserve a ZeuZ-compatible report envelope under `qa/automation/reports/zeuz-compatible/`.
11. For generated web automation, keep one portable testcase as the canonical source and run it through Playwright directly or export a Playwright spec from it.
12. When ZeuZ-compatible reports are generated, include the `zeuzExecutionLog` projection so results can map to ZeuZ Server later.

## Evidence Rule

Use ZKME artifacts before writing test cases:



Do not invent routes, selectors, credentials, database tables, mobile screens, app states, or commands. If evidence is insufficient, create a manual candidate and mark automation confidence as low.

## Hard Gate

Generated instructions are guidance; the repo-local gate is executable. For feature changes, run the QA contract verifier with the files you touched:

```bash
python qa/automation/verify_zkme_qa_contract.py --changed-file path/to/changed/file
python qa/automation/validate_test_design_quality.py
```

CI can pass a newline-delimited diff file:

```bash
git diff --name-only origin/main...HEAD > .zkme/changed-files.txt
python qa/automation/verify_zkme_qa_contract.py --changed-files-from .zkme/changed-files.txt
python qa/automation/validate_test_design_quality.py
```

If the verifier fails, add the missing manual, unit, API, DB, web, mobile, desktop, portable, or ZeuZ-compatible evidence before finishing the feature.
If the quality validator fails, deepen the manual case and testcase package until the relevant UI/API/DB/auth/audit/negative/cleanup oracles are covered.

## Final Report Requirement

Every feature final report must include:

- manual test cases created or updated
- unit tests created or updated
- E2E tests created or updated
- ZeuZ-compatible candidates created or skipped
- local execution/report output created or skipped
- Playwright direct-run/spec-export path used, when applicable
- tests run
- tests not run and why
